# 03 — CI/CD

Opinionated, cloud-agnostic defaults for build, release, and deploy. Everything
here is do-this-by-default; deviations should be justified in code review.

> Conventions: **must** = load-bearing; **should** = strong default; **prefer**
> = taste call backed by evidence; **avoid** = rejected, with a named
> replacement. See [`CONTRIBUTING.md`](../CONTRIBUTING.md).

> **One-line position.** Pipelines are *code* in the same repo as the thing
> they build, runners are *ephemeral and isolated*, deploys are *pull-based
> from Git* for mature multi-team Kubernetes and *push-based via OIDC* for
> serverless/IaC, every release artifact is **SBOM'd, signed, and attested to
> the SLSA Build level defined in ch06 §9 (currently L3 for production
> artifacts)**, and DORA's delivery+reliability metrics are the north-star
> (canonical count owned by ch11 §14) — supported by a small set of
> operational leading indicators.

---

## 1. Pipeline-as-code: pick one, commit hard

The pipeline is source code: in the same repo as the application, reviewed in
PRs, never edited in a vendor UI — click-ops in CI is click-ops in the portal.

The honest landscape for greenfield work in 2025–2026:

- **GitHub Actions** — default for most teams. Largest action ecosystem, native
  OIDC to all three major clouds, attestations and signing built in.
- **GitLab CI** — best when the SCM is already GitLab; tight integration with
  Container Registry, Dependency Scanning, Secret Detection.
- **Azure DevOps Pipelines** — still excellent for .NET/Windows. Microsoft's
  roadmap favors GitHub Actions; treat ADO as steady-state, not strategic.
- **Tekton** — when the *pipeline itself* must run on Kubernetes (regulated
  platforms, internal developer platforms). Sigstore-native but an order of
  magnitude more ops work than a SaaS runner.
- **CircleCI / Buildkite / Drone** — niche but legitimate (macOS/iOS matrices,
  hybrid self-hosted, tiny footprints).
- **Jenkins** — **avoid for greenfield.** Pipeline-as-code is bolt-on, the
  plugin trust surface is enormous, and SaaS runners removed its only real moat.

- **must:** pipelines live next to code, are reviewed in the same PR, and have
  no manual UI configuration that isn't also in the repo. *Concrete check:* a
  fresh clone + a fresh project in the CI tool reproduces the pipeline from
  `main` with no portal clicks.

## 2. Push vs pull: the deploy split

Two deploy models, chosen *per workload class*.

**Push (CI deploys).** CI runs `kubectl apply` / `terraform apply` /
`az deploy` / `aws cloudformation deploy` with short-lived OIDC credentials.
Right answer for IaC, serverless, edge/CDN, anything outside a Kubernetes
control loop, and one-shot data migrations.

**Pull (GitOps).** A controller in the cluster (**Argo CD** or **Flux**, both
CNCF Graduated) watches a Git repo of manifests and reconciles continuously.
Drift is detected and either auto-healed or surfaced; cluster credentials never
leave the cluster; `git log` is the audit log.

- **should:** for any Kubernetes environment that meets **any** ch01 §5
  trigger (>1 env, >1 cluster, >1 team sharing a cluster, or
  audit-required regime), application deploys are pull-based via Argo CD
  or Flux. (Sources: Argo CD / Flux docs; CNCF GitOps WG *OpenGitOps
  Principles v1.0*; ThoughtWorks Tech Radar — GitOps in *Adopt* since
  2022.)
- **must:** if you push-deploy to Kubernetes, the CI principal is short-lived
  (OIDC), namespace-scoped, never cluster-admin. No long-lived kubeconfig in
  repo secrets. The env repo references images by **digest** (`@sha256:…`), not
  by mutable tag — mutable tags defeat both Argo's diff and Sigstore
  verification.
- **should:** one *config repo* per environment family (or directory-per-env),
  separate from the application source repo — the boundary that lets platform
  teams promote without application-team write access.
- **avoid:** mixing models for the same workload — "GitOps for manifests but CI
  also kubectl-applies for hotfixes." That is how you get state divergence.

## 3. Supply chain: SLSA, SBOM, signing — high-maturity baseline for production critical-path workloads

Defensive baseline post-xz (CVE-2024-3094, Mar 2024): **SLSA + SBOM +
signing**, all three. Treat the full triple as the high-maturity target for
production critical-path workloads (anything internet-facing, anything with
a regulatory floor, anything whose compromise reaches paying customers); for
internal-only / pre-revenue / experimental workloads, SBOM + signing is a
defensible interim while you stand up SLSA provenance.

### 3.1 SLSA (Supply-chain Levels for Software Artifacts)

SLSA **v1.1** Build Track (current spec): **L1** = provenance exists; **L2** =
provenance generated by a hosted build service and signed
(`actions/attest-build-provenance`, GitLab SLSA provenance); **L3** = build
runs on a hardened, isolated builder with provenance non-forgeable by the
workflow itself (`slsa-framework/slsa-github-generator`, GitLab hosted SLSA
L3 runners). v1.1 formalises **VSAs (Verification Summary Attestations)** as
the consumer-facing artifact admission controllers should validate. A **Source
Track** is in active draft (`slsa.dev/spec/draft/source-requirements`) for
attesting source-side controls — review, retention, branch protection — the
same way builds are attested today.

- **must:** every release artifact (container image, binary, package) ships with a
  signed SLSA provenance attestation at the level defined in ch06 §9 (currently
  **Build L3 for production artifacts**; L2 acceptable only for non-prod).
  *Concrete check (cosign 2.x/3.x):* `cosign verify-attestation --type
  slsaprovenance --certificate-identity-regexp '^https://github.com/acme/.*'
  --certificate-oidc-issuer https://token.actions.githubusercontent.com
  <image@digest>` succeeds in a non-CI environment. (Sources: SLSA v1.1 spec
  — Build Track; GitHub *Artifact Attestations* GA Jun 2024. Owner: ch06 §9.)
- **should:** for non-production builds where L3 isn't yet feasible, document the
  L2 exception and the path to L3 (typically a switch to a hosted reusable
  workflow such as `slsa-framework/slsa-github-generator`).
- **prefer:** publish provenance via the OCI 1.1 referrers API so consumers
  fetch it alongside the image. On GitHub Actions, **Artifact Attestations**
  (`actions/attest-build-provenance@v1`, GA Jun 2024) bundle SLSA provenance
  and a Sigstore-signed in-toto attestation in one step; verify with
  `gh attestation verify <artifact> -o <org>` in CI and via the
  `github/artifact-attestations-helm-charts` policy-controller in-cluster.

### 3.2 SBOMs

Two formats matter, both standards-track. **SPDX 3.0** (Apr 2024,
profile-based — Core/Software/Build/Security/AI/Dataset/Licensing) is the
current spec; ISO/IEC 5962:2021 standardised the older **SPDX 2.2.1** and is
what most procurement contracts (EO 14028, CISA SBOM minimum elements) still
cite — emit 2.3 alongside 3.0 until consumers catch up. **CycloneDX 1.6**
(ECMA-424, Jul 2024) added **CBOM** (cryptographic BOM) and **CDXA**
(attestations); 1.7 (Oct 2025) is the final 1.x. Pick CycloneDX as primary
for security tooling (VEX, CBOM, CDXA), emit SPDX alongside for compliance.

- **must:** every container image and release binary has an SBOM generated at
  build time (Syft, `docker buildx … --sbom`, `Microsoft.Sbom.Targets`, Trivy)
  and stored *with the artifact* as an OCI referrer (`cosign attach sbom`, OCI
  1.1), not in a side bucket that drifts.
- **should:** SBOMs are scanned in CI **and continuously** post-release (Grype,
  Trivy, Dependency-Track). New CVEs against an old image must page someone.
- **avoid:** generating an SBOM by listing `package.json` / `*.csproj`. That is
  a manifest, not an SBOM — it misses transitive pins, native deps, and
  base-image content.

### 3.3 Signing: cosign keyless and KMS as peers

Two signing models are first-class. **Sigstore keyless** (Fulcio + Rekor +
cosign; CNCF **Graduated** Nov 2024, Incubating from Sept 2022; public-good
services GA Oct 2022)
issues short-lived certs bound to an OIDC identity and records every signature
in a public transparency log — best fit for OSS, public-internet SaaS, most
enterprise greenfield. **KMS/HSM-backed signing** (cosign with AWS KMS, Azure
Key Vault, GCP KMS, YubiHSM, corporate PKI) is required for FIPS boundaries,
air-gapped builds, government/defense, and threat models that can't tolerate
publishing artifact metadata externally. Pin **cosign ≥2.2** (3.x recommended
— Oct 2025, Rekor v2 client). cosign 2.0 was a deliberate break: `verify`
requires `--certificate-identity[-regexp]` + `--certificate-oidc-issuer`,
signing by tag is discouraged (sign by digest), SCT verification is on by
default. **Rekor v2** (tile-backed log, GA Oct 2025) is the new default.

- **must:** every release artifact is **signed**, and the deploy target
  verifies the signature **and** the SLSA provenance predicate before running
  it, keyed on the expected signing identity — not just "a signature exists."
  Pick: **Sigstore policy-controller** (`ClusterImagePolicy` + `TrustRoot`,
  simplest if all-in on cosign; resolves tags to digests at admission) or
  **Kyverno `verifyImages`** (multi-tool — Cosign + Notary, also
  `mutateDigest`/`verifyDigest` for digest pinning and `attestations` for
  in-toto predicates like SLSA provenance and SBOM). Connaisseur and Ratify
  are also valid; one per cluster. (Sources: Sigstore *policy-controller*;
  Kyverno *verifyImages*.)
- **should:** prefer Sigstore keyless when the build's OIDC identity (workflow +
  ref + repo) is the truth you want to bind to and public Rekor entries are
  acceptable.
- **prefer:** KMS/HSM-backed signing for private, regulated, or air-gapped
  contexts, and for any signing identity that must outlive a single workflow run.
- **avoid:** mixing models per artifact without a documented policy.

### 3.4 Registries, promotion, hermeticity

- **must:** artifact registries are **immutable** at the digest level. Tag
  overwrites disabled or audited; deletes restricted; retention policy-driven.
  (Sources: OCI Distribution Spec; AWS ECR / Azure ACR / GCP Artifact Registry
  immutable-tag docs.)
- **must:** promotion is **promote-by-digest**. The same `image@sha256:…` that
  passed staging is what runs in prod — **no rebuild on promote**. A rebuild
  produces a new digest, invalidating prior attestations, scans, and approvals.
  The signing/provenance/admission chain that gives this digest its meaning
  is owned by ch06 §9–§11 (SLSA → Sigstore → admission verification).
- **should:** builds aim for **hermeticity** — declared inputs only, no network
  at build time beyond a pinned mirrored proxy, deterministic outputs where the
  toolchain allows (`SOURCE_DATE_EPOCH`, Bazel, Nix). Hermeticity is the
  precondition for meaningful SLSA L3+ provenance.
- **prefer:** a single internal registry mirroring all upstream base images and
  language packages. Direct pulls from Docker Hub / npm / PyPI / NuGet at build
  time are a supply-chain and rate-limit liability.

### 3.5 Secrets scanning and push protection

- **must:** repository-wide **secret scanning** is enabled on every repo
  (GitHub Advanced Security, GitLab Secret Detection, or `trufflehog` /
  `gitleaks` in CI for self-hosted SCMs); findings page on-call. **Push
  protection** blocks commits containing recognized secret patterns at
  `git push` time — history rewrites do not unring the bell, the secret must
  be rotated. Custom patterns are added for internal token formats. (Sources:
  GitHub *About secret scanning*; GitLab *Secret Detection*.)

## 4. Pinned actions, OIDC scoping, and the `permissions:` block

### 4.1 Pin actions to a SHA

- **must:** every third-party action / shared template is pinned to a **full
  commit SHA**, not a tag. Tags are mutable; a compromised maintainer can
  re-point `v3` overnight (the `tj-actions/changed-files` incident, March
  2025). Use `pinact` or StepSecurity's `secure-workflows` as a CI guard;
  Renovate / Dependabot to bump SHAs with the human-readable tag in the PR
  body. *Concrete check:*
  `grep -E 'uses: [^@]+@v?[0-9]' .github/workflows/` returns zero matches for
  non-`actions/*` actions. (Sources: GitHub *Security hardening for GitHub
  Actions* — "Using third-party actions"; OpenSSF Scorecard
  `Pinned-Dependencies`.) Prefer third-party actions published as **Immutable
  Releases** (signed OCI artifacts in `ghcr.io/actions/...` with attached
  attestations, GH preview 2025); SHA-pin + pinact stays the baseline for any
  action not yet immutable.

### 4.2 OIDC: scope on the cloud side, not in the workflow

GitHub Actions, GitLab, and Buildkite all expose **OIDC claims** (`sub`,
`repository`, `workflow_ref`, `ref`, `environment`, `job_workflow_ref`, …). The
workflow proves *what it is*; the cloud's trust policy decides *what it can do*.
There is no generic "scope this OIDC token to one job" knob — scoping happens by
writing the trust policy against specific claims.

- **must:** no static cloud credentials in CI. Use **OIDC federation** to AWS (IAM
  role with web-identity trust policy), Azure (Workload Identity Federation on a
  UAMI or App Registration), or GCP (Workload Identity Federation with attribute
  mapping). Long-lived `AWS_SECRET_ACCESS_KEY`, `AZURE_CLIENT_SECRET`, or
  service-account JSON keys in repo secrets are findings. (Sources: GitHub *About
  security hardening with OpenID Connect*; AWS *Configure OIDC for GitHub Actions*;
  Azure *Workload identity federation*; Google Cloud *Workload Identity Federation
  with GitHub Actions*.)
- **must:** the cloud trust policy validates the `sub` claim (and `aud` where
  supported) against an explicit pattern that bounds **repo + ref/environment**.
  Canonical sub-claim schema and per-cloud trust-policy details (AWS / Azure /
  GCP) live in **ch06 §3**. GitHub-specific: a job's default `sub` uses
  `environment` > `ref` precedence (e.g.,
  `repo:acme/payments:environment:prod` for env-bound jobs,
  `repo:acme/payments:ref:refs/heads/main` otherwise); `pull_request`-shaped
  subjects only appear when the repo opts into a custom subject-claim template.
- **should:** for high-blast-radius roles (prod write, IAM mutation, KMS use),
  additionally constrain **`job_workflow_ref`** to a specific reusable workflow
  pinned to a tag or SHA — e.g.,
  `acme/.github/.github/workflows/deploy.yml@refs/tags/v3.2.1`. This binds the
  credential to a *reviewed* workflow. Use GitHub **deployment environments**
  (required reviewers, wait timers, branch policies) and key the trust policy
  off `environment:prod`. Also constrain `runner_environment=github-hosted`
  (rejects self-hosted-runner abuse paths); on GitHub Enterprise, pin `iss` to
  the enterprise-scoped OIDC issuer URL. (Sources: GitHub *Customizing the
  token claims*; Google Cloud *Best practices for using workload identity
  federation*.)
- **avoid:** wildcard `sub` patterns like `repo:acme/*:*` or trust policies
  that only check `iss` and `aud`. They federate the entire org.

### 4.3 `permissions:` and privileged triggers

- **must:** every workflow declares an explicit top-level `permissions:` block,
  default `contents: read`, elevating only per-job. The org/repo default for
  `GITHUB_TOKEN` should be **read** — "write-all" is the single most common
  cause of PR-triggered token theft. Treat `pull_request_target` and
  `workflow_run` workflows as privileged: they run with write tokens against
  PR code.
- **should:** **OpenSSF Scorecard** runs on a schedule and the score is visible
  in the README (target ≥7 public, ≥8 artifact-publishing).

### 4.4 GitHub Apps over PATs

- **must:** automation that acts on the SCM (open PRs, comment, manage
  releases, dispatch workflows) authenticates as a **GitHub App** (or GitLab
  Group/Project Access Token bound to a service account), not a user PAT. Apps
  have per-repo install scopes, fine-grained permissions, short-lived
  installation tokens, and survive the human leaving. (Sources: GitHub *About
  apps* / *About installation access tokens*; GitLab *Project access tokens*.)
- **should:** if a PAT is unavoidable, use a **fine-grained PAT** scoped to one
  repo and one permission, ≤90-day expiry, owner in `CODEOWNERS`.
- **avoid:** classic PATs with `repo` scope, and any PAT named
  "service-account-bot" owned by a human account.

## 5. Branch protection / Rulesets / required checks

On GitHub, use **Repository Rulesets** (the modern replacement for classic
branch protection — apply across branches by pattern, org-manageable, with a
bypass-actor audit trail).

**Baseline (must).** Every protected branch:

- Required PR review (≥1 approver, dismiss stale approvals on new commits).
- Required status checks (build, test, lint, supply-chain checks from §3,
  action-pinning lint from §4.1) — *required to pass*.
- No force-push; no deletion.
- Admins do **not** bypass — bypass-actor list is empty or limited to a
  break-glass group whose use pages security.
- *Sources:* GitHub branch-protection / rulesets docs; OpenSSF Scorecard.

**Governance (should).** Add as the team and repo mature: **signed commits** on
release branches; **linear history** or a **merge queue** (GitHub Merge Queue,
GitLab Merge Trains, Bors-NG) for repos with >5 active contributors;
**Dependabot** *or* **Renovate** with grouped updates and auto-merge for patch
updates that pass CI (pick one); **OpenSSF Scorecard** badge in README;
**Required CODEOWNERS review** for `/.github/`, IaC, and supply-chain config
(empty `CODEOWNERS` silently passes; populate before requiring).

- **avoid:** required status checks that aren't actually required to pass (the
  "required but not enforced" footgun on classic protection — Rulesets fixes
  this; verify after migration).

## 6. Monorepo vs polyrepo CI

CI strategy is the second-largest cost center after the runner fleet. Pick selective
execution from day one.

- **should:** in a monorepo, builds and tests are **affected-target scoped** — only
  what the diff touches (and its dependents) runs. Use the build system's own graph
  (Bazel `bazel query` / `rdeps`, Nx `affected`, Pants `--changed-since`, Turborepo,
  `dotnet` solution filters) rather than hand-maintained path filters. (Sources:
  Bazel *Query how-to*; Nx *Affected* docs.)
- **should:** in a polyrepo, each service repo owns its full CI graph and
  depends on **versioned shared workflows** (see §10), not on a central
  pipeline that fans in. The **merge queue** in a monorepo runs the
  affected-target subset, not the full graph; full-graph runs happen on a
  schedule and on release tags.
- **prefer:** path-based triggers (`paths:` / `paths-ignore:`) only as a
  backstop — they miss reverse deps and under-test shared-code refactors.
- **avoid:** "monorepo with one giant workflow that runs every job." Cost,
  queue time, and flaky-test rate balloon; developers learn to ignore CI.

## 7. Progressive delivery: flags, canary, blue/green, shadow

Progressive delivery **decouples deploy from release**. Deploy = bits on the
server. Release = users see them. The toolkit, ordered by complexity:

1. **Feature flags** (LaunchDarkly, Unleash, OpenFeature, homegrown table).
   Cheapest; **default** for any user-visible change. Write code against the
   **OpenFeature SDK** for your language (CNCF Incubating since Dec 2023;
   stable 1.x SDKs in Java, .NET, Node, Go, Python), not a vendor SDK.
2. **Blue/green.** Two identical environments, switch the router. For stateful
   apps where canary is hard. Cost: 2× steady-state infra.
3. **Canary.** Shift a small % of *production traffic* to the new version,
   watch SLOs, ramp or roll back. **Argo Rollouts** or **Flagger** automate
   against Prometheus / Datadog SLOs. For backend services with strong
   observability, a flag-gated dark launch often gives better regression
   signal than a weighted-traffic canary; reserve weighted canary for changes
   flags can't gate (runtime/image/kernel upgrades, schema-coupled rollouts).
4. **Shadow / mirrored traffic.** Send a copy of prod traffic to the new
   version; *don't* return its responses. Mandatory for risky read-path
   migrations. Don't shadow write paths without idempotency keys.

- **must:** every production deploy has an automated rollback path (re-deploy
  previous digest, flip blue/green, `argo rollouts undo`). Manual `kubectl
  rollout undo` typed at 3am is not a rollback path.
- **should:** canary analysis is **SLO-driven** (error rate, latency,
  saturation), not "wait 10 minutes and hope."
- **prefer:** Argo Rollouts on Argo CD; Flagger on Flux or with a service mesh
  doing the traffic split.
- **avoid:** "canary" that is actually just "deploy to one pod and see if it
  crashes." That is a smoke test.

### 7.1 Schema-coupled releases

When an application change requires a coupled schema change, the schema work
itself (expand/contract, dual-write, backfill, switch reads, drop the old)
is owned by **ch08 §10**; this chapter only covers the **release-side
coordination**:

- **must:** ship the schema change and the app change as **independently
  deployable** steps. No deploy that requires app and DDL to land
  atomically — that is unrollback-able.
- **must:** gate the cut-over with a feature flag (per §7) so reads/writes
  of the new shape can be enabled, paused, or reverted without a redeploy.
- **should:** dual-deploy old and new app versions during the contract
  step; keep both compatible with the intermediate schema.
- **must:** define the abort path before merging the migration PR — which
  flag flips off, which app version is rolled forward, what data
  reconciliation runs.
- **should:** encode the abort criteria as an **Argo Rollouts
  `AnalysisTemplate`** / **Flagger `MetricTemplate`** (replication lag,
  dual-write divergence, new-path error budget), not a runbook — the rollback
  fires on the same SLOs the canary is gated on.

## 8. Metrics: DORA as north-star, plus operational leading indicators

DORA now publishes **five** delivery+reliability metrics; ch11 §14 is the
canonical statement of the count and current names — this chapter inherits.

DORA is the *outcome* layer; pair it with a few **leading indicators** for
pipeline health, since lead time degrades long before deployment frequency does.

- **must:** the DORA metrics defined in ch11 §14 are instrumented and visible
  to the team that owns the service — not "the platform team has a dashboard
  somewhere." Trend, not snapshot — weekly average over a 4-week window.
  (Sources: DORA *Accelerate State of DevOps Report* 2023 / 2024,
  https://dora.dev/ ; *Accelerate*, Forsgren/Humble/Kim, ch. 2.)
- **should:** track operational leading indicators — **CI queue time** (p50/p95)
  for runner saturation, **pipeline duration** (p50/p95) for feedback latency,
  **flaky-test rate** (tests passing on retry without code change — rises before
  lead time and change-failure rate do), and **mean time to green** on `main`
  after a break.
- **avoid:** vanity metrics (PR count, LOC, story points, composite
  "productivity" scores) — they optimise the wrong thing (DORA 2023/2024; SPACE).

## 9. Pipeline hygiene and runner hardening

### 9.1 Hygiene

- **must:** runners are **ephemeral**. Every job starts on a fresh
  VM/container; nothing persists between runs except an explicit cache.
  (Sources: GitHub *Security hardening for self-hosted runners*; GitLab
  *Security considerations for runners*.)
- **must:** caches are **per-branch with fallback to default branch**, scoped
  by lockfile hash. Cross-PR cache poisoning is a real attack class — never
  accept a cache from an untrusted PR branch into a privileged workflow.
- **should:** **fail fast** (lint/type-check before tests, tests before
  integration, integration before deploy) and parallelize at the test-runner
  level, not by spawning more pipeline jobs.
- **avoid:** secrets in env vars echoed by any step. Mask, scope to the
  smallest job; better, fetch from a secret manager via OIDC at job start.

### 9.2 Self-hosted runner hardening

If you must self-host (cost, GPU, network reach), the threat model flips:
untrusted PR code can run on infrastructure you own. Treat runners like prod.

- **must:** runners are **just-in-time and single-use** — provisioned per job,
  destroyed after, image wiped between runs (Actions Runner Controller, GitLab
  autoscaling runners, Buildkite-agent autoscaler). Never reuse a runner VM
  across workflows. (Sources: GitHub *Autoscaling with self-hosted runners*;
  ARC *Ephemeral runners*.)
- **must:** self-hosted runners are **never attached to public-repo workflows**
  by default. Public fork PRs can run arbitrary code; self-hosted + public =
  compromise.
- **must:** **runner groups / tag scoping** bound which repos and environments
  target which runners. The prod-deploy runner group is reachable only from the
  prod-deploy reusable workflow.
- **must:** **network egress is constrained** through a known proxy / NAT, with
  allowlists for the registry, package mirrors, and cloud control planes
  actually needed. Default-allow egress is how exfiltration happens.
- **must:** **no shared Docker socket** in job containers and **no privileged
  containers** for untrusted workloads. Use rootless Docker, Kaniko, BuildKit
  rootless, or `buildah`. A mounted `/var/run/docker.sock` is root on the host.
- **must:** **no co-located secrets** on the runner host filesystem — no
  long-lived cloud keys, kubeconfigs, registry credentials, or signing keys.
  Secrets arrive per-job via OIDC and expire with it.
- **should:** runner host images are **rebuilt and re-rolled on a schedule**
  (weekly minimum), not patched in place. The image is in IaC and goes through
  the same supply-chain pipeline as applications. Host telemetry (exec, egress,
  writes outside the job workspace) ships to the same detection pipeline as
  prod.

## 10. The "platform team owns the pipeline" anti-pattern

Conway's Law applies to pipelines. If platform writes every team's pipeline,
you have built a queue. The right model: **platform** owns reusable workflows,
the runner fleet, the signing/SBOM/OIDC building blocks, and policy.
**Delivery teams** own their `.github/workflows/*.yml` / `gitlab-ci.yml`,
consuming shared blocks; they can fork the paved road and pay the cost.
Platform's product is *the paved road*, not *the only road*.

- **avoid:** a single monorepo of pipeline YAML maintained by platform and
  applied via include/extends to every service. That is the queue.
- **prefer:** versioned reusable workflows
  (`uses: org/.github/.../workflow.yml@<sha>`); service teams pin and upgrade
  on their own cadence.
